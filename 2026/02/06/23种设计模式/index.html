1.三种工厂模式的区别总结

模式 工厂数量 负责创建 主要用途
简单工厂（Simple Factory） 1 个 所有产品 工厂集中控制创建逻辑
普通工厂（Factory Method） 多个 每个工厂对应一种产品 新增产品时无需改原工厂类，只需新增工厂类
抽象工厂（Abstract Factory） 多个 一系列相关产品 创建产品族（如 CPU + 主板）

5.原型模式（Prototype）

核心思想：

通过**==复制==已有对象（==原型==）**来==创建新对象（所以叫创建型）==，而不是通过
new 或复杂构造函数创建。

对象的结构、初始状态由原型定义，克隆操作实现快速、可定制的实例生成。

关键点：

  • 接口：定义 clone() 方法，返回自身的深拷贝。

  • 目的：避免重复构造复杂对象，提高创建效率。==快！省资源！==

  • 应用场景：

    对象创建开销大（如含大量配置或外部资源）。

    需要保存多种预设模板并动态复制。

  • 优点：

    对客户端透明地创建复杂对象。

    易于动态扩展原型种类。

  • 缺点：

    深拷贝实现复杂，需关注资源与指针。

6.单例模式（Singleton Pattern）

核心思想：

保证**==一个类在系统中只有一个实例==**，并提供一个==全局访问点==。（静态的，不需要对象，直接通过类名调用）

关键点：

  • 控制实例数量：构造函数私有化，禁止复制与赋值。

  • 提供全局访问点：通过静态函数 instance() 获取唯一实例。

  • 线程安全实现：

    C++11 起：

    static 局部变量初始化保证线程安全（Meyers 单例）。

    复杂场景可用

std::call_once。

应用场景：

  • 全局资源管理（如配置中心、日志器、驱动实例）。

  • 控制访问唯一硬件接口、注册表或连接池。

优点：

  • 全局唯一、延迟创建。

  • 管理共享资源一致性。

缺点：

  • 增加全局状态，可能影响单元测试与可扩展性。

结构要点：

  • Singleton → 私有构造 + 静态访问函数 + 禁止拷贝

7.生成器/构建器模式（Builder）

目的：

将一个复杂对象的**==构建过程与其表示分离==**，使得==同样的构建过程==可以创建==不同的表示==。（比如放入不同的菜炒出不同的菜）

关键角色：

  1. Builder（抽象建造者）

：定义创建产品各部分的抽象接口。

  1. ConcreteBuilder（具体建造者）

：实现 Builder 接口，具体构造和装配产品的部件。

  1. Director（指挥者）

：调用建造者的步骤方法，控制产品构建过程。

  1. Product（产品类）

：表示被创建的复杂对象。

适用场景：

  • 对象由多个部件组成且创建过程复杂。

  • 创建过程固定，但产品的具体表示不同。

优点：

  • 使构建代码与产品表示分离。

  • 可复用构建流程创建不同对象。

2.结构型设计模式 7

1.适配器模式（Adapter）

核心思想：

适配器模式（Adapter
Pattern）的核心是将==一个类的接口==转换成客户==希望的另一个接口==，使原本接口不兼容的类能够一起工作。

它是一种结构型设计模式，主要解决”接口不匹配”问题。

适配器模式包含以下角色：

  1. Target（目标接口）：客户期望的接口。

  2. Adaptee（被适配者）：已有接口，但与目标接口不兼容。

  3. Adapter（适配器）：将 Adaptee 的接口转换成 Target 的接口。

2.桥接模式（Bridge）

核心思想：

桥接模式（Bridge
Pattern）的核心是**==将抽象部分与实现部分分离，使它们可以独立变化==**。

它是一种结构型设计模式，主要解决”多维度变化”的问题，比如抽象层次与实现层次都可能变化时避免类爆炸。

  1. Abstraction（抽象类）：定义高层抽象接口，包含对实现部分的引用。

  2. RefinedAbstraction（扩展抽象类）：在抽象接口基础上扩展。

  3. Implementor（实现接口）：定义实现部分的接口。

  4. ConcreteImplementor（具体实现类）：实现 Implementor 接口。

3.组合模式（Composite Pattern）

核心思想：

组合模式（Composite
Pattern）的核心是将对象组合成==树形结构==以表示”整体–部分”的==层次结构==，使得客户对单个对象和组合对象的使用具有一致性。

它是一种结构型设计模式，解决”单个对象与组合对象操作不一致”的问题。

  1. Component（抽象构件）：定义叶子和容器的公共接口。

  2. Leaf（叶子节点）：表示树的最小单位，没有子节点。

  3. Composite（容器节点）：可以包含子节点，实现对 Component 的统一操作。

  4. Client（客户端）：通过 Component 接口统一处理单个对象和组合对象。

4.装饰模式（Decorator Pattern）

核心思想：

装饰模式（Decorator
Pattern）的核心是在不改变原有类结构的前提下，==动态地给对象添加新的功能==。

它是一种结构型设计模式，用于替代继承实现功能扩展，使功能增强更灵活。

  1. Component（抽象构件）：定义对象的接口，可以被动态扩展。

  2. ConcreteComponent（具体构件）：被装饰的原始对象，提供基本功能。

  3. Decorator（装饰抽象类）：持有一个 Component 对象引用，并实现与 Component
    相同的接口。

  4. ConcreteDecorator（具体装饰类）：在原有功能上动态扩展新行为。

5.外观模式（Facade Pattern）

核心思想：

外观模式（Facade
Pattern）的核心是为子系统中的==一组接口==提供==一个统一的高层接口==，使子系统更易使用。

它是一种结构型设计模式，主要解决系统复杂度过高、接口繁杂的问题。

  1. Facade（外观类）：对外提供统一接口，封装内部子系统调用。

  2. Subsystem Classes（子系统类）：实现具体功能，外部通过 Facade 访问。

  3. Client（客户端）：仅与 Facade 交互，不直接访问子系统。

6.享元模式（Flyweight Pattern）

核心思想：

享元模式（Flyweight
Pattern）的核心是通过共享对象来减少内存消耗，适用于存在大量相似对象的场景，提供大量==细粒度==对象==共享元对象==。

它是一种结构型设计模式，主要解决对象过多导致内存开销大的问题。

  1. Flyweight（享元接口）：定义可共享对象的通用接口。

  2. ConcreteFlyweight（具体享元类）：实现享元接口，存储内部状态（可共享）。

  3. UnsharedConcreteFlyweight（非共享享元类）：不共享的特殊对象。

  4. FlyweightFactory（享元工厂）：创建和管理共享对象，保证相同内部状态对象共享。

  5. Client（客户端）：维护外部状态，使用享元对象。

7.代理模式（Proxy Pattern）

核心思想：

代理模式（Proxy Pattern）的核心是为某个对象==提供一个代理==对象，以控制对该对象的访问。==隔离==

它是一种结构型设计模式，主要用于访问控制、延迟加载、远程访问、性能优化等场景。

  1. Subject（抽象主题）：定义真实对象和代理的公共接口。

  2. RealSubject（真实主题）：实际执行业务逻辑的对象。

  3. Proxy（代理）：持有 RealSubject
    的引用，在调用前后进行附加操作（如权限、日志、缓存等）。

  4. Client（客户端）：通过 Subject 接口访问对象，不关心是否为代理。
